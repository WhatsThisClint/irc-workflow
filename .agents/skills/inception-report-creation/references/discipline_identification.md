# Discipline Identification — Construction Method

Same underlying method as `inception-report-review`'s disciplinary lens file, run forward: instead of inferring discipline from an existing report's content, infer it from the bare question.

## Step 1: Read the question for its phenomenon type
- **Behavioral/institutional** (governance, cess payment, cooperative management, compliance) → institutional economics, political ecology, public administration.
- **Biophysical/remote sensing** (NDVI, crop water use, source attribution) → agricultural remote sensing, hydrology, agronomy.
- **Economic/livelihoods** (willingness to pay, income effects, market access) → applied/development economics.
- **Mixed** (e.g. "does direct government funding improve WUC management" — combines institutional design with a performance/management outcome) → name both the institutional lens and whatever discipline defines "well-managed" in this context, and flag the integration point.

## Step 2: Name the discipline specifically
Not "social science" — "institutional economics of collective action" or "comparative water governance." Not "science" — "agricultural remote sensing" or "irrigation hydrology." Specificity here determines which literature and methods get pulled into Steps 2-4 of the main workflow.

## Step 3: Note what that discipline will hold the report to
State briefly (1-2 lines) the rigor standards this discipline's own reviewers would apply — construct validity and positionality for institutional/social work; measurement uncertainty and ground-truth validation for remote sensing/biophysical work — so the rest of the report is built to survive that standard, not just to sound plausible. If the field is unfamiliar or highly specialized, web-search its actual rigor norms before proceeding rather than guessing.

## Step 4: Flag interdisciplinary integration risk early
If two disciplines are both load-bearing (e.g. an institutional question answered partly with remote-sensing data on infrastructure condition), say so now — this integration point should get explicit attention in the conceptual model (Step 2 of main workflow) rather than being smoothed over.
